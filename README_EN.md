# project_template

A cross-platform C++ CMake project template supporting **Linux (GCC/Clang)** and **Windows (MSVC/Clang/MinGW)** multi-platform development.

## Features

- ✅ **Cross-platform Build** — Supports Linux GCC/Clang + Ninja and Windows MSVC/Clang/MinGW + Ninja/Visual Studio
- ✅ **Auto-generated Presets** — One-click environment detection, automatically generates a tailored `CMakePresets.json`
- ✅ **Multiple Target Types** — Examples for executables, static libraries (`.a`/`.lib`), and shared libraries (`.so`/`.dll`)
- ✅ **CTest Integration** — Built-in test support, run `ctest` directly after `cmake --build`
- ✅ **CPack Packaging** — One-click packaging into `.tar.gz` (Linux) or `.zip` (Windows)
- ✅ **Workflow Preset** — Complete configure → build → test → package pipeline with a single command
- ✅ **Sanitizer Support** — Optionally enable AddressSanitizer / UndefinedBehaviorSanitizer / ThreadSanitizer
- ✅ **Code Formatting** — Built-in `.clang-format` (Google style)
- ✅ **Debug Friendly** — Debug mode generates full debug information with optimizations disabled
- ✅ **Multi VS Version Detection** — Automatically detects all installed Visual Studio versions on Windows (vswhere)
- ✅ **Multi Compiler Support** — Simultaneously detects MSVC, Clang-cl, standalone Clang, MinGW GCC
- ✅ **Cross-platform Unified Experience** — Same preset generation scripts on Linux/Windows, consistent behavior
- ✅ **VS Code Deep Integration** — Works with CMake Tools extension, auto-detects Presets, visual compiler selection, build, test, package, install

## Quick Start

### Step 1: Customize the Project (Optional)

Before using the template, it is recommended to modify the project name and version to suit your needs:

**`CMakeLists.txt`** — Modify the project name and version:
```cmake
project(YourProjectName VERSION YourVersion)
```

**`cmake/CPackPreset.cmake`** — Modify the package description and vendor name:
```cmake
set(CPACK_PACKAGE_DESCRIPTION_SUMMARY "Your application description")
set(CPACK_PACKAGE_VENDOR "Your company name")
```

> The sub-projects (`src/project1`, `src/staticLib1`, etc.) are example templates; feel free to add, modify, or remove them as needed.

### Step 2: Generate Presets

`CMakePresets.json` contains environment-specific configurations such as compiler paths, which vary between machines. When using the project for the first time, run the auto-detection script:

```bash
# Linux
./generate_presets.sh

# Windows
generate_presets.bat
```

The script automatically detects:
- **Operating System** — Linux / Windows
- **Compilers** — GCC, Clang, MSVC and their versions
- **Build Tools** — Ninja, Unix Makefiles, Visual Studio

After generation, you can view the available Presets:

```bash
cmake --list-presets
```

### Step 3: Configure and Build

All the following commands can also be performed through the VS Code + CMake Tools graphical interface.

```bash
# Configure and build (use actual names from --list-presets output)
# Linux example: GCC 14.2 Debug
cmake --preset gcc_14.2-debug
cmake --build --preset gcc_14.2-debug

# Windows example: MSVC 17
cmake --preset msvc17
cmake --build --preset msvc17-debug

# Windows example: standalone Clang 20.1 Debug
cmake --preset clang_20.1-debug
cmake --build --preset clang_20.1-debug

# Run (Linux)
./build/gcc_14.2-debug/bin/project1d
./build/gcc_14.2-debug/bin/TestLibd

# Run (Windows)
build\msvc17\bin\Debug\project1d.exe
build\msvc17\bin\Debug\TestLibd.exe

# Test
ctest --preset ctest-gcc_14.2-debug

# Package
cpack --preset cpack-gcc_14.2-debug

# Install
cmake --install build/gcc_14.2-debug --prefix install/gcc_14.2-debug

# One-click workflow
# Configure → Build → Test → Package, all in one command:

cmake --workflow --preset workflow-gcc_14.2-debug

```

### Using in VS Code

After installing the **CMake Tools** extension, it will automatically read the Presets from `CMakePresets.json`. You can select from the status bar at the bottom of VS Code:

- **Kit** — Select compiler (corresponds to configurePreset)
- **Build** — Select build configuration
- **Run CTest** — Run tests
- **Package** — Package

## CMakePresets.json Details

`CMakePresets.json` is automatically generated by `cmake/GeneratePresets.cmake` and defines the complete build lifecycle Presets, covering five stages: **Configure → Build → Test → Package → Workflow**.

### Preset Naming Rules

The auto-generated Preset names follow this format:

| Platform | Configure Preset Name | Description |
|----------|----------------------|-------------|
| Linux GCC | `gcc_{version}-debug` / `gcc_{version}-release` | e.g., `gcc_14.2-debug` |
| Linux Clang | `clang_{version}-debug` / `clang_{version}-release` | e.g., `clang_20.1-debug` |
| Windows MSVC | `msvc{major_version}` | e.g., `msvc17`, multi-config, specify Debug/Release at build time |
| Windows Clang | `clang_{version}-debug` / `clang_{version}-release` | e.g., `clang_20.1-debug`, standalone Clang |
| Windows MinGW | `gcc_{version}-debug` / `gcc_{version}-release` | e.g., `gcc_14.2-debug`, MinGW GCC |

Other Preset types (build, test, package, workflow) are derived from the configure Preset name:

| Preset Type | Naming Format | Example |
|------------|--------------|---------|
| buildPreset | `{configure_name}` | `gcc_14.2-debug` |
| testPreset | `ctest-{configure_name}` | `ctest-gcc_14.2-debug` |
| packagePreset | `cpack-{configure_name}` | `cpack-gcc_14.2-debug` |
| workflowPreset | `workflow-{configure_name}` | `workflow-gcc_14.2-debug` |

### Configure Presets (configurePresets)

Configure Presets define CMake configuration parameters, including generator, compiler, build type, and output directory.

- **Single-config generators (Ninja / Unix Makefiles)**: Used for standalone Clang, GCC (cross-platform). One Preset each for Debug/Release, build type specified at configure time via `CMAKE_BUILD_TYPE`
- **Multi-config generators (Visual Studio)**: Used for MSVC, Clang-cl (Windows only). One Preset containing both Debug/Release, build type specified at build time

### Build Presets (buildPresets)

Build Presets are associated with their corresponding configure Presets.

- **Single-config**: Directly references the configure Preset name
- **Multi-config**: Generates one each for Debug/Release (e.g., `msvc17-debug`, `msvc17-release`)

### Test Presets (testPresets)

Test Presets are associated with their corresponding configure Presets and are used to run CTest.

### Package Presets (packagePresets)

Package Presets are associated with their corresponding configure Presets and are used for CPack packaging.

- **Linux**: Packages as `.tar.gz`, output to `packages/` directory
- **Windows**: Packages as `.zip`, output to `packages/` directory

### Workflow Presets (workflowPresets)

Workflow Presets chain together **Configure → Build → Test → Package** into a single command:

```bash
cmake --workflow --preset workflow-gcc_14.2-debug
```

This is equivalent to running:

```bash
cmake --preset gcc_14.2-debug
cmake --build --preset gcc_14.2-debug
ctest --preset ctest-gcc_14.2-debug
cpack --preset cpack-gcc_14.2-debug
```

## Build Options Details

### Debug Mode

- **Debug Info**: `-g3` (GCC/Clang) / `/Zi` (MSVC) — Most detailed debug information
- **Optimization**: `-O0` (GCC/Clang) / `/Od` (MSVC) — Optimizations disabled for easier debugging
- **Frame Pointers**: Frame pointers preserved for better stack traces
- **Suffix**: Executables automatically get a `d` suffix (e.g., `project1d`)

### Release Mode

- **Optimization**: Highest optimization level enabled by default
- **Optional**: Debug info or disabled optimization can be enabled for Release mode via CMake variables

### Sanitizer (Optional)

Enable via CMake variables at configure time:

```bash
# Enable AddressSanitizer (detect memory errors)
cmake --preset gcc_14.2-debug -DENABLE_FSANITIZE_ADDRESS=ON

# Enable UndefinedBehaviorSanitizer (detect undefined behavior)
cmake --preset gcc_14.2-debug -DENABLE_FSANITIZE_UNDEFINED=ON

# Enable ThreadSanitizer (detect data races)
cmake --preset gcc_14.2-debug -DENABLE_FSANITIZE_THREAD=ON
```

All variables are OFF by default.

> **Note**: Sanitizers have significant performance overhead and are recommended for Debug mode only.
